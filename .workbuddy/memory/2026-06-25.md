### 第二次修复：音色名不匹配导致 TCP 挂死
**LOG 分析**：Python 服务启动成功，但 generate-tts 后只有 "generate: text.len=..." 行，没有 "合成成功/失败" 输出。说明 TCP 请求发出后 Python 服务没返回响应。

**根因**：`.dev-user-data` 的 DB 里 `doubao_tts_voice` 插件的 voiceList 是**旧的短名格式**：
- `name: "zh_female_cancan"`（缺少 `_uranus_bigtts` 后缀）
- `config.speaker: "zh_female_cancan"`（也是错的，应该是 `zh_female_cancan_uranus_bigtts`）
- 只有 2 个音色（xiaohe / cancan），缺少 vv/云舟/小天/苏菲

前端把 `zh_female_cancan` 传给 Python 服务 → Python 传给 ByteDance API → API 不认识短名 → WebSocket 挂死 → TCP 空超时（60s）→ 前端永远无响应。

**修复**：
1. 直接更新 `.dev-user-data` DB：voiceList 改为 6 个音色全名（`*_uranus_bigtts`），config 改为 camelCase。
2. main.js generate-tts 加强 voiceName 查找逻辑：先精确匹配 → 再尝试补 `_uranus_bigtts` 后缀 → 都找不到才用默认。防御旧 DB 残留。

**验证**：Python 服务接受 `zh_female_cancan`（短名）时 TCP 挂死，接受 `zh_female_cancan_uranus_bigtts`（全名）时 1.19s 合成成功。

### 第三个 bug：切换声音后播放就停了
**现象**：用户切换豆包音色后，朗读直接卡住不播。
**根因**：`handleVoiceSwitch` 第 498 行设了 `TTSUtil.isPaused = true`（TTSUtil 类静态属性），但后续只通过 `this.setState({ isPaused: false })` 改组件状态，**TTSUtil.isPaused 永远没被改回 false**。
然后 `handleCustomRead` → `cacheAudio`，cacheAudio 第 172 行检查 `this.isPaused`（即 TTSUtil.isPaused）= true → 直接 break → 一段音频都没缓存 → 播放停。
**修复**：在 `handleVoiceSwitch` 调 `handleCustomRead` 前显式 `TTSUtil.isPaused = false`（component.tsx 第 528 行附近）。
**注意**：原代码 `TTSUtil.isPaused = true` 是为了终止旧的 cacheAudio 循环，这个意图是对的，但少了对应的复位。setAudioPaths 不改 isPaused（语义是清缓存，不应改暂停态）。

### 第四个 bug："在此处朗读"卡顿 + 快速点击失败（竞态条件）
**现象**：用户用"在此处朗读"时卡住，快速点击会出现失败。
**根因**：`TTSUtil.stopAudio()` 用了 `setTimeout(1000)` 延迟清理 `audioPaths`。快速点击"在此处朗读"时，第一次 stop 的 1 秒计时器在新会话的 `cacheAudio` 过程中触发，**清空新会话刚缓存的音频路径** → `readAloud` 找不到音频 → `loaderror` → 失败。
此外，旧 `cacheAudio` 的 `getAudioPath` 返回后仍会 push 过期数据到新会话的 `audioPaths`，旧 `handleCustomRead` 循环也不会因 `isAudioOn` 变化而退出（React 批处理把 false→true 合并，旧循环看到的始终是 true）。

**修复（三层防护）**：
1. **ttsUtil.ts `stopAudio`**：移除 `setTimeout(1000)`，改为立即清理；设 `this.player = null`。
2. **ttsUtil.ts `sessionId` 机制**：新增 `static sessionId` 计数器，`stopAudio`/`setAudioPaths` 递增。`cacheAudio` 开头捕获 `mySession`，在 `getAudioPath` 返回后、`Promise.all` 批次完成后均检查 `sessionId !== mySession` → 丢弃过期结果并 return。防止旧 cacheAudio 的数据污染新会话。
3. **component.tsx `readGeneration` 机制**：新增实例属性 `readGeneration`，`handleStartAudio`/`handleVoiceSwitch`/`handlePrevSentence`/`handleNextSentence` 均递增。`handleCustomRead` 开头捕获 `myGen`，在循环顶部、`cacheAudio` 返回后、`handleSpeech` 返回后检查 `readGeneration !== myGen` → 旧循环安静退出（不报错、不 setState，避免把新会话也停了）。
4. **component.tsx `handleSpeechAutoStartRequest`**：在 `handleStop` 后、`handleStartAudio` 前显式 `TTSUtil.isPaused = false`；`handleStartAudio` 改为 `await`（原来是 fire-and-forget）。
5. **ttsUtil.ts `setAudioPaths`**：也重置 `isPaused = false`（原来不重置）。

### 第五个 bug：豆包 TTS 语速不生效
**现象**：UI 上选语速对豆包 TTS 无效，始终用 1.0x 合成。
**根因**：main.js doubao 分支发给 Python 服务的 TCP 请求只有 `{ text, voice, format, sample_rate }`，**没传 speed**。Python `make_payload` 的 `audio_params` 也没 `speed_ratio` 字段。speed 变量在 main.js 第 1010 行被解构但从未使用。
**修复**：
1. main.js: 将前端传入的 `speed`（已做 `speed*100-100` 转换）还原为倍速比 `speedRatio = (speed + 100) / 100`，限制 [0.2, 6.0]，加入 TCP 请求 `speed_ratio` 字段。加 NaN 防御。
2. doubao_tts_server.py: `make_payload` 的 `audio_params` 加 `speed_ratio`；`synthesize_text` 接受并透传 `speed_ratio`；`handle_client` 从请求读 `speed_ratio`，加 TypeError 防御 + 范围限制。

**构建验证**：`npm run build` 编译成功（+150 B），Python `py_compile` 语法检查通过。

## 2026-06-26 倍速参数名修复（第六个 bug）

### 根因
豆包双向流式 TTS API（`wss://openspeech.bytedance.com/api/v3/tts/bidirection`，resource_id: `seed-tts-2.0`）的语速参数名是 **`speech_rate`**（取值 [-50, 100]，100=2.0x，-50=0.5x，0=1.0x），不是 `speed_ratio`。之前用了错误的参数名 `speed_ratio`（倍速比格式），API 直接忽略，导致所有语速都是 1.0x。

### 关键发现
- 前端 `handleCustomRead` 做 `speed * 100 - 100` 转换后的值，**格式和 `speech_rate` 完全一致**（-50=0.5x, 0=1.0x, 100=2.0x），不需要做 `(speed+100)/100` 还原为倍速比
- API 限制范围 [-50, 100]（即 0.5x~2.0x），超出范围会被 clamp

### 修复
1. **main.js**：移除 `speedRatio = (speed+100)/100` 转换，直接用 `speechRate = Number(speed) || 0`，clamp 到 [-50, 100]，TCP 请求字段从 `speed_ratio` 改为 `speech_rate`
2. **doubao_tts_server.py**：`make_payload` 的 `audio_params.speed_ratio` → `speech_rate`；`synthesize_text` 参数 `speed_ratio` → `speech_rate`；`handle_client` 读取 `speech_rate`，clamp [-50, 100]；START_SESSION 的 payload 调用也改了
3. 重新 PyInstaller 打包 exe + 重新 electron-builder 打包便携版

### 验证
端到端测试确认 speech_rate 生效：
- speech_rate=-50: 音频 10.04s (0.5x)
- speech_rate=0: 音频 5.18s (1.0x)
- speech_rate=50: 音频 3.45s (1.5x)
- speech_rate=100: 音频 2.59s (2.0x)
